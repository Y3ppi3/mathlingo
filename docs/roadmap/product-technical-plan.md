# MathLingo — план продукта и технической архитектуры (R1–R4)

Автор роли: Product Manager / Solution Architect.
Основа: аудит текущего кода на момент написания (`mathlingo-backend/app/models.py`,
`app/routes/*`, `app/auth.py`, `main.py`, `mathlingo-frontend/src/components/admin/*`,
`alembic/versions/*`). План строится как расширение существующей системы, а не
переписывание с нуля — везде, где это возможно, помечено, что уже есть и что
нужно добавить.

## 0. Текущее состояние (baseline, важно для реализма плана)

Что уже есть и на что план опирается:

- **Backend**: FastAPI + SQLAlchemy + Alembic + PostgreSQL. Модели: `Admin`,
  `User`, `Achievement`, `AdventureMap`, `MapLocation`, `TaskGroup`, `Task`,
  `UserProgress`, `Subject`. Роли пользователей — **плоские**: `User` и
  `Admin` без градации прав; любой `Admin` может всё. Бутстрап первого
  админа — регистрация без авторизации, пока в БД нет ни одного админа
  (`app/routes/admin.py:register_admin`).
- **Контент**: `Task` — уже общая модель задания (title, description, subject,
  subject_id, difficulty_level: int, reward_points, estimated_time_seconds),
  но **без** статуса публикации, версии, источника (ручной/AI) и уровня в
  смысле «базовый/стандартный/продвинутый» (сейчас `difficulty_level` —
  плоское целое число).
- **Прогресс**: `UserProgress` хранит только **агрегаты** на пользователя
  (`current_level`, `total_points`, JSON-списки локаций/ачивок). **Нет
  таблицы попыток** (attempt-level history) — ни по обычным заданиям, ни по
  играм. Это ключевой пробел для R2 (адаптивность) и R3 (аналитика).
- **Auth/CSRF**: JWT в httponly/secure/samesite=strict cookie + Bearer для
  админки (`app/auth.py`). CSRF-мидлварь в `main.py` — double-submit токен,
  но хранится **в памяти процесса** (`csrf_tokens = {}`), что ограничивает
  деплой одним воркером. Зафиксировано также в
  `.claude/skills/secure-coding/SKILL.md`.
- **Alembic**: в рабочей копии история миграций схлопнута до одной
  `6240db818db1_initial_migration.py` (остальные версии удалены, но ещё не
  закоммичены). План исходит из того, что это будет зафиксировано как новая
  база **до** начала миграций R1.
- **Frontend admin**: уже есть `admin/{GamificationPanel,SubjectForm,
  SubjectsPanel,TaskForm,TasksPanel,UsersPanel}.tsx` — то есть базовая
  админка **частично существует**. R1 не строит админку с нуля, а
  реорганизует существующие панели в зоны «God Mode» + добавляет RBAC,
  статусы и аудит.
- **Игры**: `DerivFall.tsx` уже принимает `problemsSource` как проп (не
  хардкод) — задел под шаблонизацию в R3 уже частично сделан.
- **Санитизация**: `mathlingo-frontend/src/utils/sanitizeHtml.ts`
  (DOMPurify) уже существует и используется в `TaskSolver.tsx`/`DerivFall.tsx`
  — переиспользуется в R2 для HTML-санитизации AI-контента.

## 1. Сквозные основания (действуют во всех релизах)

Чтобы не повторять в каждом релизе одно и то же, здесь фиксируются базовые
контракты, которые R1 вводит, а R2/R3 используют без изменений.

### 1.1 Модель ролей (RBAC)

Роль хранится как колонка на стороне staff-пользователей (`Admin.role`,
enum), а не на стороне обычных `User`:

| Роль | Значение | Может |
|---|---|---|
| `superadmin` | владелец системы | всё: роли, квоты, настройки, необратимые операции (архивация с последующим hard-delete, смена ролей, запуск AI) |
| `content_manager` | контент-оператор | создание/редактирование/публикация контента, AI-заказы, игровые сценарии; **не** может менять роли/квоты/системные настройки |
| `teacher` | преподаватель | просмотр учащихся и прогресса, проверка черновиков контента, возврат на доработку (`needs_revision`); **не** может публиковать/архивировать/удалять |

Проверка роли — **всегда на backend** (FastAPI dependency
`require_role(*roles)` поверх существующего `get_admin_current_user`),
скрытие пунктов меню на фронтенде — только UX-слой, не граница доверия.

### 1.2 Статусная модель контента

Единый статус-контракт для любого учебного контента (ручного и AI):

```
draft → in_review → approved → published
                 ↘ needs_revision ↗
published → archived
```

- `archived` заменяет `delete` для контента и пользователей (см. требование
  «удаление → архивирование»). Настоящее удаление — только для
  superadmin, отдельное необратимое действие с подтверждением, не часть
  обычного жизненного цикла.
- Уровень темы (`basic|standard|advanced`) — атрибут контента, ортогональный
  статусу.

### 1.3 Аудит

Единая таблица `audit_log(id, actor_admin_id, actor_role, action,
entity_type, entity_id, before_json, after_json, created_at, request_id)`.
Пишется через общий backend-декоратор/dependency на всех мутирующих
эндпоинтах админки — не точечно, иначе аудит быстро станет неполным
(это явный DoD-критерий каждого релиза).

## RELEASE 1 (4–6 недель) — Единая контентная модель, уровни/статусы, RBAC, аудит, базовая админка

### 1. Цели и границы scope

**Цель**: превратить плоскую пару `User/Admin` и сырой `Task` в систему с
ролями, статусами публикации и аудитом, поверх которой R2/R3 строят
адаптивность и игры. Это releaseфундамент — без него AI-конвейер (R2) и
геймификация шаблонов (R3) некуда «публиковать».

**В scope**:
- Роли `superadmin/content_manager/teacher` + миграция текущих админов.
- Расширение `Task` → «контент-айтем» (статус, уровень, версия, источник).
- Новая сущность `Skill` (тема внутри `Subject`), т.к. «уровень
  пересчитывается по теме», а не по предмету целиком.
- Аудит-лог + middleware записи.
- Реорганизация существующих admin-панелей в зоны God Mode; заглушки для
  зон, зависящих от R2/R3 (AI-очередь, игры и карты, аналитика качества)
  с понятным «недоступно до R2/R3».
- Массовые операции над контентом (публикация/архивация/статус) и
  пользователями (смена статуса), импорт/экспорт CSV/JSON.

**Вне scope** (сознательно, чтобы не размыть 4–6 недель):
- Диагностика, адаптивность, AI-генерация — это R2.
- Конструктор игровых сценариев — R3.
- Назначение контента ученикам/когортам (`assign`) — вводится в R1 как
  ручное точечное назначение (MVP, без адаптивной логики), полноценная
  «рекомендованное на основе уровня» версия — R2, когда появится
  mastery-модель. Явно фиксирую это допущение, т.к. в задаче «назначение
  контента» упомянуто без привязки к релизу.

### 2. Пользовательские сценарии

1. **Superadmin** логинится → создаёт учётку `content_manager` → назначает
   роль → просматривает аудит-лог созданной учётки.
2. **Content-manager** создаёт новую тему (`Skill`) внутри существующего
   `Subject` → создаёт задание вручную → сохраняет как `draft` → отправляет
   в `in_review`.
3. **Teacher** открывает очередь `in_review` → просматривает черновик →
   либо `approve` (переход к content-manager для публикации, т.к.
   publish — право content_manager/superadmin, не teacher), либо
   возвращает в `needs_revision` с комментарием.
4. **Content-manager** публикует одобренный айтем (`published`) → выполняет
   массовую архивацию устаревшего набора заданий по фильтру (тема + уровень)
   → выполняет экспорт каталога в CSV.
5. **Superadmin** архивирует пользователя вместо удаления → в audit-log
   видна запись с `before/after`.
6. Любой staff-пользователь открывает `overview` — видит нерабочие зоны
   (AI-очередь и т.д.) с пометкой «доступно с R2».

### 3. Изменения frontend, backend, БД и API

**БД (новые/изменённые таблицы)**:
- `admins.role` (enum: `superadmin|content_manager|teacher`, NOT NULL,
  default выставляется миграцией для существующих строк).
- Новая таблица `skills(id, subject_id FK, name, code, order, is_active)`.
- `tasks` (расширение): `level` (enum `basic|standard|advanced`, default
  `standard`), `status` (enum, default `published` для существующих строк —
  см. миграции ниже), `version` (int, default 1), `source` (enum
  `manual|ai`, default `manual`), `skill_id` (FK, nullable на первое время),
  `created_by_admin_id`, `approved_by_admin_id`, `published_at`,
  `archived_at`. Поле `subject` (string legacy) **не удаляется** в R1 —
  помечается deprecated в схеме/докстринге, план удаления — отдельная
  задача после того, как весь фронтенд перейдёт на `subject_id`/`skill_id`.
- `audit_log(...)` — см. §1.3.
- `content_status_history(id, task_id, from_status, to_status, actor_admin_id,
  comment, created_at)` — нужен teacher-сценарию «вернуть с комментарием».

**Backend**:
- `app/models.py`: перечисленные колонки/таблицы.
- `app/auth.py`: `require_role(*roles)` dependency поверх
  `get_admin_current_user`; `role` кладётся в JWT payload при логине.
- Новый роутер `app/routes/skills.py` (CRUD тем).
- `app/routes/tasks.py` / `admin.py`: эндпоинты статус-переходов
  (`POST /admin/tasks/{id}/submit-review`, `/approve`, `/publish`,
  `/request-changes`, `/archive`) с проверкой допустимых переходов и роли.
  Массовые операции: `POST /admin/tasks/bulk` (`{ids, action}`).
- Общий audit middleware/dependency, подключаемый декларативно к мутирующим
  роутам.
- Импорт/экспорт: `GET /admin/tasks/export`, `POST /admin/tasks/import`
  (CSV/JSON, валидация схемой перед вставкой — переиспользует schema-валидацию,
  которая в R2 понадобится для AI-конвейера, поэтому пишется как отдельная
  переиспользуемая функция уже сейчас).

**Frontend**:
- Реструктуризация `admin/*` в зоны God Mode (роутинг
  `/admin/overview`, `/admin/content`, `/admin/ai-queue` (заглушка),
  `/admin/games` (заглушка), `/admin/students`, `/admin/quality` (заглушка),
  `/admin/users`, `/admin/audit`).
- `TasksPanel.tsx`/`TaskForm.tsx` расширяются под статус/уровень/переходы
  (кнопки действий видны по роли — но сервер всё равно перепроверяет).
  `TaskForm.tsx` уже рендерит контент задания через `dangerouslySetInnerHTML`
  где-то в предпросмотре — обязательно завести через существующий
  `sanitizeHtml()`.
- `UsersPanel.tsx`: смена статуса (архивация) вместо удаления, массовые
  действия.
- Новая `AuditLogPanel.tsx` — таблица с фильтрами по actor/entity/действию.

### 4. Миграции данных

1. Зафиксировать текущий сквош Alembic-истории как коммит **до** первой
   R1-миграции (иначе новые миграции лягут поверх незакоммиченного
   состояния и историю будет невозможно восстановить у других
   разработчиков).
2. `0002_add_admin_roles`: добавить `role`, backfill — **все существующие
   админы получают `superadmin`** (безопасный дефолт: понижать роль вручную
   проще и безопаснее, чем случайно оставить кого-то без прав).
3. `0003_add_skills_table` + backfill: создать по одному `Skill` "General"
   на каждый существующий `Subject`, привязать существующие `tasks.skill_id`
   к нему (не оставлять NULL, иначе фильтрация по теме в UI ломается сразу).
4. `0004_add_content_status_fields`: добавить `level/status/version/source/
   created_by_admin_id/approved_by_admin_id/published_at/archived_at` в
   `tasks`; backfill — `status='published'`, `level='standard'`,
   `source='manual'`, `version=1` для всех текущих строк (текущие задания уже
   находятся в проде фактически «опубликованы»).
5. `0005_add_audit_log_and_status_history`.
6. Все миграции — с рабочим `downgrade()` (проверяется в CI, см. §6).

### 5. Модель ролей и разрешений

| Действие | superadmin | content_manager | teacher |
|---|---|---|---|
| Создать/редактировать draft | ✅ | ✅ | ❌ |
| Submit to review | ✅ | ✅ | ❌ |
| Approve / Request changes | ✅ | ❌* | ✅ |
| Publish | ✅ | ✅ | ❌ |
| Archive | ✅ | ✅ | ❌ |
| Массовые операции над контентом | ✅ | ✅ | ❌ |
| Управление ролями/квотами | ✅ | ❌ | ❌ |
| Просмотр учащихся/прогресса | ✅ | ✅ (read) | ✅ |
| Просмотр аудита | ✅ | частично (свои действия) | ❌ |
| Необратимые операции (hard delete) | ✅ (с подтверждением) | ❌ | ❌ |

\* content_manager не аппрувит свой же контент по принципу
разделения обязанностей (four-eyes) — можно ослабить решением продукта
позже, но по умолчанию запрещено.

### 6. Тестовая стратегия

- **Backend (pytest, расширение `tests/test_auth.py`)**: unit-тесты на
  каждый статус-переход (валидные/невалидные), permission-матрица — для
  каждого эндпоинта таблица «роль → ожидаемый код ответа» (401/403/200),
  тесты аудит-лога (мутация → запись создана с корректным before/after).
- **Migrations**: CI-шаг `alembic upgrade head && alembic downgrade -1 &&
  alembic upgrade head` на чистой БД — ловит необратимые миграции сразу.
- **Frontend (vitest)**: рендер-тесты статусных переходов в `TasksPanel`
  (кнопки скрыты/задизейблены по роли), тест на то, что предпросмотр
  контента идёт через `sanitizeHtml`.
- **E2E (ручной чек-лист на этом этапе, автоматизация — с R3)**: полный
  цикл draft → in_review → approve → publish → bulk archive под тремя
  ролями.

### 7. Риски, зависимости и security-требования

- **Риск**: бутстрап-эндпоинт `register_admin` без авторизации (пока БД
  пуста) остаётся потенциальной дырой в non-prod окружениях, если
  БД когда-либо очищается автоматически. Требование: задокументировать и,
  если нужно, закрыть флагом окружения `ALLOW_BOOTSTRAP_ADMIN`.
- **Риск**: CSRF-токены в памяти процесса (см. `SKILL.md`) — пока
  1 воркер, не блокер, но с ростом админ-нагрузки (больше staff-пользователей
  одновременно) это должно быть закрыто **до** масштабирования на несколько
  воркеров/подов. Фиксирую как security-требование R1: не увеличивать
  `--workers` в проде, пока это не вынесено в Redis/БД.
- **Зависимость**: R2 (mastery-модель) и R3 (аналитика) напрямую зависят от
  `skill_id` — если в R1 сущность `Skill` спроектирована плохо (например,
  без иерархии подтем), переделка в R2 будет дорогой. Ревью схемы `Skill`
  — обязательный чекпоинт перед мержем миграции `0003`.
- **Security**: RBAC-проверки — только server-side; фронтенд-скрытие кнопок
  не считается защитой (проверяется отдельным security-тестом:
  прямой запрос от teacher-токена на `/publish` должен вернуть 403).
  Изменения проходят по чек-листу `.claude/skills/secure-coding/SKILL.md`.

### 8. Definition of Done

- Все backend/frontend тесты и `alembic upgrade/downgrade` зелёные в CI.
- `tsc --noEmit`, `eslint`, `pytest` без ошибок.
- Полный цикл контента (draft→published→archived) и управление
  ролями/аудитом выполняется через UI **без единого изменения кода или
  прямых правок в БД** — контрольный прогон делает не автор фичи.
- RBAC permission-матрица покрыта тестами на 100% мутирующих эндпоинтов.
- Аудит-лог фиксирует 100% мутирующих admin-действий (проверяется тестом,
  не вручную).
- Security-чеклист по `secure-coding` skill пройден (CSRF/cookies/CORS/
  XSS/SQL — без регрессий).

### 9. Очерёдность реализации по задачам

1. Alembic: зафиксировать сквош истории → `0002` (roles) → миграция бэкенд-
   зависимостей (`require_role`).
2. `0003` (skills) + CRUD `skills.py` + UI раздела тем.
3. `0004` (content fields) + статус-переходы в `tasks.py`/`admin.py`.
4. `0005` (audit) + middleware + `AuditLogPanel.tsx`.
5. Массовые операции + импорт/экспорт.
6. Реструктуризация admin-роутинга в зоны God Mode + заглушки будущих зон.

### 10. Метрики внутреннего тестирования

- % мутирующих эндпоинтов, покрытых RBAC-тестами (цель: 100%).
- % мутирующих эндпоинтов, пишущих в audit_log (цель: 100%).
- Время прохождения ручного E2E-сценария (все 3 роли) — базовая метрика
  для сравнения с R2/R3.
- Количество дефектов по severity, найденных на internal QA, до релиза.
- Успешность `upgrade→downgrade→upgrade` миграций в CI (бинарно, блокирующее).

## RELEASE 2 (4–6 недель) — Диагностика, модель освоения, адаптивность, AI-конвейер

### 1. Цели и границы scope

**Цель**: закрыть главный продуктовый разрыв — сейчас в БД нет ни одной
записи о попытке решения (только агрегаты в `UserProgress`). R2 вводит
attempt-level трекинг, диагностику по темам и пересчёт уровня, а также
конвейер AI-генерации заданий поверх контентной модели R1.

**В scope**: диагностика по темам, `mastery_state` и пересчёт, UI причины
рекомендации + временный выбор соседнего уровня, AI-заказы и весь конвейер
до публикации, пост-публикационный мониторинг AI-контента, квоты/аудит AI.

**Вне scope**: выбор конкретного AI-провайдера и политика передачи данных —
явный decision gate (см. §7), конвейер строится против
провайдер-агностичного интерфейса + мок-провайдера, чтобы не блокировать
разработку. Игровые шаблоны/конструктор — R3 (но game-attempts уже пишутся
в общую таблицу попыток, введённую здесь, чтобы R3 не переделывала схему).

### 2. Пользовательские сценарии

1. **Ученик** проходит диагностику по конкретной теме (не общий тест) →
   видит стартовый уровень с объяснением («точность 40%, среднее время
   решения выше нормы для стандартного уровня») → может временно выбрать
   соседний уровень.
2. Система пересчитывает уровень ученика по теме после очередного окна
   попыток (accuracy, время, использование подсказок) — **не** после
   единичной попытки.
3. Игровые результаты (когда появятся в R3) участвуют в пересчёте только
   агрегированно (то же скользящее окно, отдельный вес источника
   `game` vs `task`, не 1:1 с обычной попыткой) — контракт таблицы попыток
   закладывается в R2 так, чтобы R3 просто начала писать в неё.
4. **Content-manager** создаёт AI-заказ: тема, навык, уровень, тип задания
   (single-answer/multiple-choice), количество, ограничения → отслеживает
   статус пакета в AI-очереди.
5. Заказ проходит: генерация → черновик → schema validation → HTML
   sanitization → детерминированная проверка мат. ответа → AI-критик →
   попадает в очередь **teacher/content_manager** ручной проверки (как
   обычный `in_review` из R1) → публикуется вручную, никогда автоматически.
6. После публикации **teacher** видит агрегаты по AI-заданию (точность,
   время, пропуски, жалобы) и может пожаловаться → аномальные показатели
   автоматически возвращают айтем в повторную проверку.
7. **Superadmin** настраивает квоту AI-генераций на content_manager'а и
   видит расход квоты в аудите.

### 3. Изменения frontend, backend, БД и API

**БД**:
- `attempts(id, user_id, content_type[task|diagnostic|game], content_id,
  skill_id, is_correct, time_spent_ms, hints_used, source[manual|game],
  created_at)` — центральная новая таблица, от которой зависят и R2, и R3.

**Backend**:

- AI pipeline как явная state machine (`ai_pipeline.py`): каждая стадия —
  отдельная чистая функция с результатом в `ai_generation_items`, что даёт
  трассируемость по каждому этапу отдельно (требование задачи).
  `AIProviderClient` — интерфейс с мок-реализацией на первом этапе; реальный
  провайдер подключается только после решения decision gate.
- Детерминированная проверка мат. ответа — переиспользуемый модуль (нужен
  и для AI-конвейера, и потенциально для будущей проверки ручных заданий).
- Квоты: dependency, проверяющая `ai_quota` перед созданием заказа, 429 при

- Экран рекомендации уровня с секцией «почему» (структурированные факторы

- Зона «AI-очередь» в God Mode (была заглушкой в R1): форма заказа,

- Зона «аналитика качества» (была заглушкой): точность/время/пропуски/
  жалобы по AI-заданию, список аномалий.

### 6. Тестовая стратегия

- **Алгоритм адаптивности**: набор фикстур «последовательность попыток →
  ожидаемый уровень» (golden tests), включая граничные случаи (мало данных,
  резкий скачок точности, много подсказок при верном ответе).
- **AI pipeline**: контрактные тесты на каждую стадию отдельно (валидный/
  невалидный JSON от «генератора», HTML с XSS-пейлоадом должен быть
  вычищен санитайзером, мат. ответ с эквивалентными, но синтаксически
  разными выражениями — регресс-набор для детерминированной проверки).
  Отдельный тест: **AI не может опубликовать сам себя** — эндпоинт publish
  для `source=ai` требует review-record с `reviewed_by != null`.
- **Нагрузочный тест** пересчёта mastery (батч на N пользователей ×
  M тем) — чтобы ночная job укладывалась в окно.
- **Security**: HTML-санитизация AI-черновиков тестируется тем же
  набором XSS-пейлоадов, что и ручной контент (переиспользуем логику из
  secure-coding skill).

### 7. Риски, зависимости и security-требования

- **Decision gate (закрыт)**: выбор AI-провайдера и политика передачи данных
  **приняты** — см. `docs/decisions/r2-ai-provider-decision.md`. Провайдер:
  Anthropic Claude Sonnet 5 (резерв на сложный контент — Opus 4.8), доступ и
  оплата через прокси-агрегатора с поставщиком в ЕС (юрлицо-биллинг —
  российское, прямой доступ у Anthropic/OpenAI/Google для таких юрлиц
  заблокирован). До подключения реального вызова в `AIProviderClient`
  конвейер по-прежнему работает на мок-провайдере — переключение на боевой
  вызов остаётся отдельной задачей реализации, не частью этого решения.
- **Риск холодного старта данных**: mastery без диагностики — низкая
  уверенность рекомендаций в первые недели. Требование: UI обязан отдельно
  помечать `confidence=low` рекомендации.
- **Риск**: детерминированная проверка математического ответа — источник
  ложных срабатываний (эквивалентные выражения, погрешность плавающей
  точки, разный порядок операндов). Нужен явный golden-set до того, как
  AI-конвейер выйдет из экспериментального режима.
- **Security**: AI-заказы и квоты — обязательный аудит (кто, когда, сколько
  заказал) — переиспользует `audit_log` из R1. Content flags/жалобы не
  должны позволять анонимному ученику влиять на статус контента напрямую
  (жалоба создаёт `content_flags`, но статус меняет только человек).

### 8. Definition of Done

- Ни один AI-сгенерированный айтем не может получить `status=published` без
  записи ручной проверки — покрыто тестом, не только документацией.
  End-to-end трассировка одного заказа: prompt_template + model_version +
  входные параметры + результат каждой стадии видны в UI для reviewer'а.
- Диагностика и пересчёт уровня работают по реальным попыткам (attempts),
  адаптивный алгоритм проходит golden-tests на 100%.
- AI-provider decision gate закрыт (решение принято и задокументировано)
  до перехода конвейера в реальный (не мок) режим — это критерий, а не сама
  фича; на мок-провайдере остальной DoD может быть выполнен независимо.
- Все DoD-пункты R1 продолжают выполняться (регрессия RBAC/аудита
  проверяется тем же test suite).

### 9. Очерёдность реализации по задачам

1. `attempts` + запись попыток из существующего task-flow (без адаптивности
   ещё, просто трекинг) — минимальный риск, максимальная польза для всех
   последующих шагов.
2. `mastery_state` + сервис пересчёта + golden-tests алгоритма.
3. Диагностика по теме (UI + backend) поверх готового mastery-сервиса.
4. UI рекомендации уровня с «почему» + временный выбор соседнего уровня.
5. AI pipeline на мок-провайдере: schema validation → sanitization →
   deterministic check → критик-заглушка → ручная проверка (reuse R1
   review UI) → publish.
6. Квоты + аудит AI-заказов.
7. Пост-публикационный мониторинг + аномалии → content_flags.
8. Decision gate по провайдеру закрывается параллельно (product/legal
   трек, не блокирует пункты 1–7).

### 10. Метрики внутреннего тестирования

- % совпадения адаптивного алгоритма с ожиданиями golden-set (цель: 100%
  перед релизом, регресс — блокер).
- AI: конверсия draft → published (сколько % заказанных айтемов реально
  доходит до публикации), среднее время ручной проверки на айтем.
- % AI-айтемов, ушедших на повторную проверку по аномалии (первые недели —
  ожидаемо высокий, метрика для тюнинга порогов).
- Завершаемость диагностики в internal QA (% дошедших до конца по теме).
- Производительность батч-пересчёта mastery (сек/1000 пользователей).

## RELEASE 3 (4–6 недель) — Игровые шаблоны, конструктор сценариев, сквозная аналитика

### 1. Цели и границы scope

**Цель**: превратить 3 захардкоженных игровых компонента в шаблоны с единым
контрактом конфигурации, дать content_manager'у конструктор без кода, и
собрать финальный God Mode dashboard на данных, накопленных в R1/R2.

**В scope**: `GameConfig`-контракт, миграция DerivFall/IntegralBuilder/
MathLab на конфиг вместо хардкода, no-code конструктор сценариев, publish-
гейт (schema + тесты + предпросмотр + чек-лист), запись игровых попыток в
`attempts` (из R2), финальный dashboard.

**Вне scope**: новые игровые механики (сама логика геймплея остаётся зоной
разработки, не конструктора — контракт явно это разделяет).

### 2. Пользовательские сценарии

1. **Content-manager** открывает зону «игры и карты» → выбирает шаблон
   DerivFall → привязывает источник заданий (фильтр по теме/навыку/уровню/
   тегам из каталога R1) → настраивает таймер, попытки, награды,
   рандомизацию, порядок, доступность (даты/группы), тексты → сохраняет как
   `draft`.
2. Система прогоняет schema validation → content_manager открывает
   предпросмотр «от лица ученика» → проходит чек-лист (тексты корректны,
   нет плейсхолдеров, KaTeX рендерится) → публикует.
3. **Ученик** играет в опубликованный сценарий → попытки/точность/время/
   подсказки/завершение пишутся в `attempts` (та же таблица, что и обычные
   задания, `content_type=game`).
4. **Superadmin/content_manager** открывает финальный dashboard: активность,
   прогресс по темам, сложность и время решения, качество AI-контента (R2),
   завершаемость игр, очередь проверок, ошибки публикаций, действия
   администраторов — всё на одном экране.

### 6. Тестовая стратегия

- **E2E (впервые формализуется как автоматизация, не только чек-лист)**:

  проходится учеником → попытка попадает в attempts и виден на dashboard».
  Дополнительно — E2E snapshot/diff по три существующим играм **до и после**

- **Performance**: время ответа dashboard-агрегаций на синтетическом

### 7. Риски, зависимости и security-требования

- **Главный риск релиза**: миграция трёх уже работающих игр на общий
  контракт без регрессии. `DerivFall` — низкий риск (уже принимает
  `problemsSource`), `IntegralBuilder`/`MathLab` риск неизвестен без аудита
  их текущего кода — первая задача R3 explicitly включает этот аудит перед
  оценкой оставшейся части релиза (может сдвинуть 4–6 недель, если эти
  компоненты сильно завязаны на собственный хардкод).
- **Зависимость**: dashboard зависит от данных, накопленных в R1
  (audit_log) и R2 (attempts, mastery, ai). Если R2 задерживается, R3
  можно начинать (game-контракт и конструктор), но финальный dashboard
  будет неполным до готовности R2-данных.
- **Security**: конструктор сценариев — это, по сути, ввод произвольного
  JSON-конфига content_manager'ом; конфиг не должен допускать инъекцию
  произвольного HTML/скрипта в тексты — те же правила `sanitizeHtml`, что и
  для обычного контента, применяются к любому текстовому полю конфига,
  отображаемому в игре.

### 8. Definition of Done

- Content-manager создаёт и публикует новый сценарий целиком через UI, без
  участия разработчика — контрольный прогон делает не автор фичи (совпадает
  с общим критерием готовности).
- Все три существующие игры воспроизведены через конфиг без регрессии
  (E2E snapshot diff — чисто).
- Финальный dashboard показывает данные по всем перечисленным в задаче
  разрезам и укладывается в согласованный performance-бюджет.
- Игровые попытки видны в той же модели данных, что и обычные задания
  (проверяется тестом: попытка в игре формирует запись в `attempts` и
  влияет на `mastery_state` только агрегированно, не за одну попытку).

### 9. Очерёдность реализации по задачам

## RELEASE 4 (3–5 недель) — Гигиена репозитория, Redis-кеширование, производительность, визуальное обновление

### 1. Цели и границы scope

**Цель**: R1–R3 закрыли продуктовую функциональность; R4 — не про новые фичи, а
про то, чтобы система выдержала реальную нагрузку и не отпугивала пользователя
визуально. Три трека, которые запросил владелец продукта, оказались тесно
связаны технически, поэтому идут одним релизом: гигиена/структура репозитория
(инфраструктура для остального), Redis (кеш + исправление реальной проблемы
масштабирования), фронтенд (директории → сборка → дизайн-система → визуал).

**Аудит на момент написания** (не гипотезы, а конкретные находки в текущем коде):

- `mathlingo-backend/.env` и `mathlingo-frontend/.env` **закоммичены в git с
  первого коммита** (`DATABASE_URL`, `SECRET_KEY`, `VITE_API_URL`) — секреты в
  истории репозитория. Сейчас там dev-значения (`test123` и т.п. из
  `docker-compose.yml`), но это не повод оставлять как есть.
- Нет корневого `.gitignore` — 30 файлов `__pycache__/*.pyc` и
  `mathlingo-backend/debug.log` закоммичены и меняются в каждом diff'е.
- `app/routes/admin.py` (1123 строки) и `app/routes/gamification.py`
  (774 строки) смешивают несколько доменов каждый (задания, AI-заказы, квоты,
  content-flags — в admin.py; карты, submit-answer, диагностика, mastery,
  студенческий доступ к game-scenarios — в gamification.py), в отличие от
  уже принятого в R3 паттерна "один файл — один домен"
  (`game_scenarios.py`, `dashboard.py`).
- Нет `app/config.py` — переменные окружения читаются россыпью через
  `os.getenv(...)` по всему коду; для `REDIS_URL` и подобного понадобится
  централизованное место всё равно.
- `SQLAlchemy create_engine()` без `pool_size`/`max_overflow`/`pool_pre_ping`
  — сейчас незаметно на dev-нагрузке, станет проблемой при реальном трафике
  (обрыв «протухших» соединений, никакого контроля над числом коннектов к БД).
- **CSRF-токены хранятся в `csrf_tokens = {}` — обычном Python dict в памяти
  процесса** (`main.py`). Это не просто "негде кешировать" — это реальный баг
  масштабирования: при нескольких worker-процессах uvicorn/gunicorn токен,
  выданный одним воркером, не пройдёт проверку на другом; при рестарте
  процесса все активные сессии теряют CSRF-валидность разом. Один из немногих
  случаев, где Redis закрывает не perf-хотелку, а реальный корректностный
  дефект.
- `GET /gamification/game-scenarios/active/{template_key}` (дергается на
  каждый старт каждой игры каждым учеником) и
  `GET /admin/dashboard/overview` (R3 задача 7, несколько агрегаций на каждый
  реквест) — самые горячие и одновременно самые кешируемые эндпоинты: первый
  меняется только по publish/archive/update конкретного сценария, второй —
  сводка, для которой секунды staleness продуктово не критичны.
- Фронтенд: `mathlingo-frontend/src/api/api.ts` — мёртвый дублирующий модуль
  (ни одного импорта в коде), параллельный `utils/api.ts`. `src/utils/`
  вперемешку хранит и API-клиенты (`api.ts`, `adminApi.ts`), и утилиты
  (`sanitizeHtml.ts`, `auth.ts`) — нет разделения. `src/components/ui/`
  содержит один файл (`ProgressBar.tsx`), при этом переиспользуемые
  `Button`/`Input`/`Navbar`/`UserMenu` лежат россыпью в корне `components/`.
- Ни один route не обёрнут в `React.lazy`/`Suspense` — весь admin-бандл
  (recharts, dnd-kit) и все игровые зависимости (katex, mathjs, recharts,
  framer-motion) грузятся студентом уже на экране логина, хотя используются
  только за конкретными маршрутами.
- `vite.config.ts` содержит `test: {...}` через `defineConfig` из `vite`, а
  не `vitest/config` — источник уже встреченной в этой сессии ошибки сборки
  `'test' does not exist in type 'UserConfigExport'` (см. коммиты R3) —
  безопасный, изолированный фикс, попутно к остальной работе над сборкой.

**Вне scope**: полный редизайн/ребрендинг с нуля (нет brief'а/референсов —
если он появится, это отдельный трек поверх дизайн-системы из этого релиза);
миграция на другой стек (Next.js, другой ORM и т.п.); прод-инфраструктура
(Kubernetes, CDN, реальный Redis-кластер с failover) — это R4 закладывает
интеграцию и паттерны на уровне кода и dev docker-compose, конкретная
prod-топология — отдельная DevOps-задача вне этого документа.

### 2. Пользовательские сценарии

1. **Ученик** открывает игру — конфиг сценария приходит из кеша, а не из БД
   при каждом заходе; после того как content_manager публикует новый
   сценарий, ученик в течение секунд видит актуальную версию (инвалидация по
   publish/archive/update, не по TTL с задержкой).
2. **Content-manager/superadmin** открывает финальный dashboard (R3 задача 7)
   — агрегации отдаются из кеша с коротким TTL вместо пересчёта всех запросов
   на каждый рефреш страницы.
3. **Разработчик** (следующая сессия работы над проектом) открывает
   `app/routes/` и видит по одному файлу на домен вместо двух файлов на
   тысячу строк; открывает `.env.example`, а не гадает, какие переменные
   нужны, потому что реальный `.env` больше не в репозитории.
4. **Любой пользователь фронтенда** быстрее видит первый экран: студенческая
   часть приложения больше не тянет admin-бандл и наоборот.
5. **Ученик/content_manager** видит визуально более цельный интерфейс на
   ключевых экранах (Dashboard ученика, карта приключений, экран игры,
   авторизация) — не полный ребрендинг, а последовательное применение единой
   дизайн-системы вместо разрозненных ad hoc стилей.

Не меняется — R4 не вводит новых ролей и не переопределяет существующий

проверяется тем же test suite, что и раньше (RBAC-тесты бьют по путям, а не
по именам файлов).

### 6. Тестовая стратегия

- **Регрессия обязательна на каждом шаге разбиения файлов**: после переноса
  роутов в новые модули — полный прогон `pytest` (весь текущий suite, без
  исключений) до коммита, не после. Разбиение файла не должно менять ни один
  путь/метод/response_model — если тест ловит расхождение, это баг переноса,
  а не тест "надо поправить".
- Тесты на кеш-слой: cache hit/miss, инвалидация по каждому мутирующему
  эндпоинту (publish/archive/update сценария → следующий GET active видит
  свежие данные, не старые из кеша), TTL-истечение для dashboard.
- Тест на CSRF через Redis: токен, выданный в одном "процессе" (можно
  смоделировать двумя независимыми Redis-клиентами в тесте), проходит
  проверку в другом — то есть закрывает именно многопроцессный кейс, который
  in-memory dict не мог пройти в принципе.
- Frontend: `tsc --noEmit` и `eslint` — обязательное условие после каждого
  шага реструктуризации (см. риски о живом дереве изменений ниже);
  visual regression — ручной чек-лист "открыть каждую переименованную
  страницу, ничего не сломано визуально" (нет visual regression
  тулинга в проекте — вводить его ради одного релиза избыточно).
- **Performance-бюджет фиксируется до, не после**: время ответа
  `/admin/dashboard/overview` и `/gamification/game-scenarios/active/...`
  с кешем и без, на синтетическом объёме — конкретные числа появляются в
  начале реализации кеш-задач (пункт 4 очерёдности), не постфактум.

### 7. Риски, зависимости и security-требования

- **Главный риск — не технический, а процессный**: этот релиз — сплошная
  реструктуризация уже работающего, полностью протестированного кода
  (R1–R3). Самый вероятный способ что-то сломать — не Redis, а разбиение
  `admin.py`/`gamification.py` на файлы с потерей RBAC-зависимости или
  импорта по дороге. Смягчение: разбиение — маленькими шагами (один
  логический блок эндпоинтов за раз), полный test suite после каждого шага,
  ни один шаг не коммитится без зелёных тестов.
- **Зависимость от живой сессии**: этот документ написан после того, как
  R1–R3 уже полностью реализованы и закоммичены в этой же сессии работы —
  R4 не блокирует и не блокируется предыдущими релизами по данным (в отличие
  от R2→R3), это независимая техническая работа поверх готовой функциональности.
- **Security**: перенос `.env` из истории — `git rm --cached` убирает файл
  из будущих коммитов, но **не** из истории; если понадобится полностью
  вычистить историю (`git filter-repo`/BFG) — это отдельное, разрушительное
  действие, требующее явного согласия и координации (перезапись истории у
  всех, кто уже клонировал репозиторий). Ротация `SECRET_KEY` обязательна
  в любом случае, вне зависимости от того, чистится ли история.
- **Security (Redis)**: CSRF-токены и кеш игровых сценариев не содержат PII
  и не требуют шифрования на уровне значений; но Redis в dev-`docker-compose`
  должен слушать только внутреннюю docker-сеть (без публикации порта наружу,
  в отличие от текущего `db`, который сейчас пробрасывает `5432` наружу —
  тот же паттерн для `redis` не нужен и не должен повторяться).
- **Риск дизайн-системы без brief'а**: "более привлекательный фронтенд" без
  референса — субъективный критерий приёмки. Смягчение: дизайн-токены и
  первый экран (Login/Register — самый маленький по площади) проходят перед
  владельцем продукта как контрольная точка до того, как тот же подход
  масштабируется на Dashboard/AdventureMap/GamePage — чтобы не переделывать
  четыре экрана из-за фидбека, полученного после первого.

   `debug.log`, `.env.example` для обоих проектов. Самая дешёвая и самая
   срочная задача — делается первой, ничего от неё не зависит.
2. Backend: `app/config.py` (pydantic Settings) + перевод существующих
   `os.getenv` на него. Инфраструктура для задачи 4 (нужен `REDIS_URL`).
3. Backend: разбиение `admin.py` и `gamification.py` на модули по домену
   (см. §3), перенос game-scenario студенческих эндпоинтов в
   `game_scenarios.py`. Полный regression test suite после каждого файла.
4. Redis: сервис в `docker-compose.yml`, `app/services/cache.py`, перевод
   CSRF-хранилища на Redis (закрывает реальный корректностный баг — делается
   раньше кеширования, а не после, чтобы не тащить оба риска в одном шаге).
   Здесь же фиксируется performance-бюджет (baseline замеры без кеша).
5. Redis: кеширование `game-scenarios/active/...`, `dashboard/overview`,
   `subjects`/`skills` + инвалидация на мутирующих эндпоинтах + тесты hit/
   miss/invalidation.
6. Backend: тюнинг connection pool, `GZipMiddleware`, точечная ревизия N+1
   в дашборд-агрегациях.
7. Frontend: удаление мёртвого кода (`api/api.ts`), перенос API-клиентов в
   `src/api/`, консолидация `components/ui/`+`components/layout/`.
8. Frontend: `React.lazy`/`Suspense` по маршрутам (admin-поддерево, игровые
   компоненты), фикс `vite.config.ts` (`vitest/config`), замер размера
   бандла до/после.
9. Frontend: дизайн-токены + Login/Register как контрольная точка перед
   владельцем продукта (см. риск в §7).
10. Frontend: применение дизайн-системы к Dashboard ученика, AdventureMapPage,
    GamePage — после одобрения контрольной точки из задачи 9.
11. Финальная проверка: полный backend test suite + `tsc`/`eslint`/`vite build`
    + живой E2E на dev-стенде (в т.ч. заново — cache hit-rate, что после
    рестарта backend-контейнера кеш переживает или корректно прогревается)
    + сравнение performance-бюджета с baseline из задачи 4.

### 10. Метрики внутреннего тестирования

- Время ответа `/admin/dashboard/overview` и
  `/gamification/game-scenarios/active/{template_key}` до/после кеша на
  синтетическом объёме (число фиксируется в задаче 4, сравнивается в задаче 11).
- Cache hit-rate на горячих эндпоинтах после "прогрева" (несколько минут
  типовой нагрузки).
- Размер основного JS-бандла и admin-чанка до/после code-splitting
  (`vite build` output).
- Количество регрессий, пойманных test suite при разбиении `admin.py`/
  `gamification.py` (цель — 0 к моменту коммита каждого шага, не "в конце").

## 3. Общие критерии готовности (для всех релизов)

- **Автоматика**: backend `pytest` зелёный, `alembic upgrade`/`downgrade`
  проверены в CI, frontend `tsc --noEmit` + `eslint` + `vitest` зелёные,
  security-чеклист (`secure-coding` skill: CSRF/cookies/CORS/XSS/SQL/
  секреты) пройден на diff релиза.
- **Основной админский сценарий релиза выполняется без изменения кода** —
  контрольный прогон делает не автор фичи, а второй человек по написанному
  сценарию (это единственный способ реально проверить «no-code» обещание
  задачи, а не поверить автору на слово).
- **Успешное внутреннее end-to-end тестирование** — начиная с R1 это ручной
  чек-лист, с R3 — частично автоматизированный E2E-набор.
- **Внешний пилот не планируется** — метрики успеха всех релизов внутренние
  (дефекты, покрытие тестами, производительность), не пользовательские
  метрики вовлечённости.
