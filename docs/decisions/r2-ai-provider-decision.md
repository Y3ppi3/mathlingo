# Решение по decision gate: выбор AI-провайдера и политика передачи данных (R2, задача 8)

**Статус**: принято
**Дата**: 2026-07-14
**Закрывает**: `docs/roadmap/product-technical-plan.md`, R2 §7 "Decision gate (блокирующий)" и сводную таблицу decision gates (§2)

## Контекст

AI-конвейер (R2, задачи 5-7) разработан и протестирован на `MockAIProvider` против
провайдер-агностичного интерфейса `AIProviderClient` (`app/services/ai_provider.py`) —
это было сознательным решением именно для того, чтобы не блокировать разработку
конвейера до закрытия этого решения. Переход конвейера из мок-режима в реальный
требовал ответа на два вопроса:

1. Какой провайдер/модель использовать для генерации математических заданий.
2. Какая политика передачи данных применяется (что уходит во внешний API, куда,
   на каких условиях хранения).

Ключевое уточнение, изменившее сравнение: юридическое лицо, оплачивающее API, —
российское. Прямой биллинг у Anthropic/OpenAI/Google для российских юрлиц
заблокирован (экспортный контроль/санкции, проверяется по биллинговым данным, не
только по IP). **Решено: доступ и оплата закрываются через прокси/агрегатора с
поставщиком в ЕС** — это снимает вопрос доступа с повестки данного решения; ниже
он фигурирует только как остаточный риск, а не как блокер выбора модели.

## Рассмотренные варианты

| Провайдер | Доступ из РФ | Ориентировочная цена (per 1M токенов, in/out) | Сигнал по качеству математики | Итог |
|---|---|---|---|---|
| **Anthropic** (Claude Sonnet 5 / Opus 4.8) | Через EU-прокси | ~$2-3 / $10-15 (Sonnet 5, интро-цена) | Сильная; хорошо ложится на существующий детерминированный чекер + AI-критик в конвейере | **Выбран (Sonnet 5, основной)** |
| **OpenAI** (GPT-5.5/5.4) | Через EU-прокси | ~$2.5-5 / $15-30 | Сильная, топ-уровень | Резервный вариант при пересмотре |
| **Google** (Gemini 3.1 Pro) | Через EU-прокси | ~$2 / $12 (≤200K контекст) | Сильная | Резервный вариант при пересмотре |
| **DeepSeek** (V4-Pro/R1) | Не под экспортным контролем, но отдельный вопрос оплаты | ~$0.44 / $0.87 (V4-Pro) — на порядок дешевле | Очень сильные публичные бенчмарки (MATH-500 ~96-97) | Кандидат на пересмотр ради экономии, не приоритет сейчас — качество не было под вопросом, а операционная сложность (прокси уже есть под западных провайдеров, второй канал ради экономии — отдельная работа) не оправдана на старте |
| **GigaChat MAX** (Сбер) | Нативный, рублёвый биллинг | Низкая (от 600₽/мес + ~0.2₽/1000 токенов) | Заявлено: школьный/базовый вузовский уровень, +25% точности на математике в MAX-версии | Не выбран — раз доступ к западным провайдерам решён прокси, приоритет "качество" (см. решение пользователя) перевешивает; остаётся кандидатом, если стоимость прокси станет проблемой |
| YandexGPT | Нативный, рублёвый биллинг | Не проверялось детально | Не проверялось детально | Не рассматривался глубоко — тот же класс, что GigaChat; не потребовался при выбранном пути через EU-прокси |

## Решение

1. **Провайдер и модель**: **Anthropic, Claude Sonnet 5** (`claude-sonnet-5`) — основная
   модель для `AIProviderClient` в продовом режиме.
   - **Обоснование**: конвейер уже валидирует результат детерминированным чекером
     математического ответа и AI-критиком (`app/services/ai_pipeline.py`) до того,
     как черновик попадает к живому reviewer'у — то есть у одной генерации нет
     задачи быть безошибочной, ошибка ловится ниже по пайплайну. В этих условиях
     разумно оптимизировать по цене/качеству, а не брать самый дорогой топ-уровень
     "на всякий случай".
   - **Резерв на сложный контент**: **Claude Opus 4.8** (`claude-opus-4-8`) — как
     override на уровне конкретного `AIGenerationOrder` (поле `model_version` уже
     есть в схеме), для тем/уровней, где Sonnet 5 показывает повышенный процент
     `failed_answer_check`/`failed_validation` на golden-set (см. "Критерии
     пересмотра" ниже).
2. **Канал доступа**: через прокси/агрегатора с юрлицом-поставщиком в ЕС,
   рублёвый/безопасный для российского юрлица биллинг на своей стороне. Выбор
   конкретного агрегатора — отдельная закупочная задача, не часть этого решения.
3. **Политика передачи данных**:
   - В API уходят только: `subject_id`, `skill_id`, `level`, `task_type`,
     `count`, `constraints`, текст `prompt_template` — **без каких-либо
     персональных данных учеников**. Это уже так по конструкции
     `AIGenerationOrder` (см. `app/models.py`) и не требует изменений кода.
   - Запросить у Anthropic (через прокси-поставщика) договорные условия
     **нулевого/минимального хранения данных (zero/short data retention)** —
     насколько это в принципе доступно через прокси-канал, отдельно уточнить
     у поставщика, т.к. проксирование добавляет второй контур обработки данных
     (см. остаточные риски).
   - `model_version` и `prompt_template_id`/версия обязательно логируются в
     `ai_generation_items` на каждый заказ — так уже сделано (R2 задача 5) и не
     требует изменений.

## Остаточные риски (не блокируют закрытие gate, но требуют слежения)

- **Прокси как вторая точка отказа и второй контур данных.** Данные проходят
  через инфраструктуру EU-поставщика, а не только Anthropic — его собственная
  политика хранения/безопасности должна быть отдельно подтверждена и,
  желательно, зафиксирована договором, а не принята на слово.
- **Наценка и надёжность прокси-канала** — стоимость и SLA прокси-поставщика
  не оценивались в рамках этого решения (закупочный вопрос), могут повлиять на
  экономику при масштабировании AI-очереди.
- **Вендор-концентрация**: Claude уже используется как основной инструмент
  разработки (Claude Code) — тот же вендор теперь и в продовом контент-пайплайне.
  Осознанное упрощение (один вендор, один прокси-канал), но стоит держать в
  голове при следующем пересмотре бюджета/рисков.

## Критерии пересмотра

Пересмотреть это решение, если:
- Доля `failed_validation`/`failed_answer_check` по Sonnet 5 на golden-set
  превышает согласованный порог (порог зафиксировать отдельно, когда наберётся
  статистика по реальным заказам через `content_quality`-агрегаты, R2 §7/R3 §7
  dashboard).
- Стоимость AI-очереди на реальном объёме делает DeepSeek или GigaChat MAX
  экономически значимой альтернативой — тогда провести повторный бейк-офф на
  golden-set перед переключением.
- Прокси-канал станет ненадёжным (частые сбои/задержки) или договорные условия
  по хранению данных не удастся получить — тогда рассмотреть прямых доменных
  провайдеров (GigaChat/YandexGPT) как основной, а не резервный путь.

## Что это разблокирует

- `AIProviderClient` может подключать реальный вызов Claude Sonnet 5 вместо
  `MockAIProvider` — конвейер (validation → sanitization → deterministic check →
  AI-критик → `in_review`) не меняется, меняется только реализация клиента.
- R2 Definition of Done, пункт "AI-provider decision gate закрыт" — выполнен.
